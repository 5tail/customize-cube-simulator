# 05 派工模板：直接複製、填空、貼上

> 用法：使用者開新對話時，或（Claude Code 模式）主模型派 subagent 時，複製對應模板填空。每個模板都內建「派工三件套」：目標與動機、驗收條件、回報格式。〔〕內是填空處。
> 每個模板第一行都要求先讀 CLAUDE.md 與 handoff——這行不能省，省了就是 docs/00 洞 1。

---

## T1 搜尋／查找（找檔案、找某段邏輯在哪、查用法）

```
先讀 CLAUDE.md 與 docs/07-handoff.md。

任務：找出〔要找什麼，例：目前圖片大小限制寫在哪些地方〕。
動機：〔為什麼要找，例:準備把上限從 5MB 改成 3MB，需要知道所有要同步改的位置〕。

驗收條件：列出所有相關位置，不能只列第一個就停。
回報格式：每個位置一行「檔案路徑:行號 — 一句話說明」。不要貼大段程式碼。
不要做：只找，不改任何檔案。
```

## T2 實作（新功能）

```
先讀 CLAUDE.md、docs/07-handoff.md，以及 docs/01 中〔對應 Phase〕的規格。

任務：實作〔功能名稱與一句話描述〕。
動機：〔這是 Phase 幾的哪個項目〕。

範圍：只動〔目錄/檔案範圍〕。明確不做：〔例：不動後台、不加新依賴〕。
技術邊界：只用 docs/02 白名單內的技術。需要白名單外的東西時停下來問我。

驗收條件（全部要達成，對照 docs/04 R2）：
1. 〔具體可觀察的行為，例：上傳 6MB 圖片會被擋下並顯示中文提示〕
2. CI 綠燈
3. 附至少 1 個測試
回報格式：改了哪些檔案（路徑:行號）＋ 人工驗收三步驟 ＋ 更新 handoff。
```

## T3 重構（不改行為，只改結構）

```
先讀 CLAUDE.md 與 docs/07-handoff.md。

任務：重構〔範圍〕，目的是〔例：把重複三次的上傳驗證邏輯抽成共用函式〕。
動機：〔為什麼值得做〕。

鐵律：重構 = 外部行為零改變。開始前先確認現有測試綠燈；結束後同一批測試必須仍是綠燈，且不新增功能、不改 UI 文案。
改動前：對要動的每個檔案留備份副本（cp 檔名 檔名.bak，完成驗收後才刪 .bak）。
範圍上限：最多動〔N〕個檔案。發現需要動更多 → 停下來回報，不要繼續。

驗收條件：測試前後皆綠、diff 裡沒有行為變更、.bak 已清理。
回報格式：改動摘要（每檔一行）＋ 測試結果 ＋ 更新 handoff。
```

## T4 研究（查外部資料、比較做法、寫決策建議）

```
先讀 CLAUDE.md 與 docs/02-architecture.md。

任務：研究〔問題，例：Turnstile 在純前端站的正確接法〕。
動機：〔要支撐哪個決定〕。

要求：
1. 優先官方文件，其次高票社群解答；每個結論附來源連結。
2. 區分「查證過的事實」與「你的推論」，推論要標明。
3. 查不到就寫查不到，不准編造 API 或參數名。
結論格式：建議做法（一段）＋ 替代方案與代價（各一句）＋ 來源清單。
產出寫進〔docs/research-主題.md〕，對話裡只給三行摘要。
不要做：只研究不實作。
```

## T5 審查（驗證不自驗的執行工具）

```
你是審查者，這是全新對話，你沒有寫這些程式碼的包袱——這正是找你來的原因。
先讀 CLAUDE.md、docs/01、docs/02、docs/04。

審查對象：〔檔案清單或 PR 連結〕，它宣稱完成了〔任務描述與驗收條件〕。

依序檢查並逐項回答：
1. 驗收條件是否每一條都真的達成？（對程式碼，不對說明文字）
2. docs/04 R5 品質底線五項逐條過。
3. 有沒有偷加規格外的東西（對照 docs/01「明確不做」）？
4. 有沒有弱模型常見錯誤：寫死的密鑰、沒處理的錯誤路徑、只在 happy path 能動？
回報格式：「通過 / 有條件通過 / 退回」＋ 問題清單（每項：嚴重度、檔案:行號、一句話修法）。
鐵律：只審不修。你動手修 = 下一輪又要找別人審你。
```

---

## 通用附註
- 模板可以微調字句，但**三件套（目標動機、驗收條件、回報格式）與第一行讀檔指令不可刪**。
- 任務同時符合兩型（例：邊研究邊實作）→ 拆成兩次派工，先 T4 後 T2。一次對話塞兩型是失焦的開始。
