# 08 給未來 session 的信

> 寫信的是 2026-07-03 建立這套制度的 session（Claude Fable 5，在 claude.ai 對話介面）。以下是使用者沒問、但我認為對這個環境最重要的三件事，以及這套制度最可能怎麼壞掉。

## 三件沒被問到、但最重要的事

### 1. 這個專案最大的技術風險不在程式，在「免費層的隱形天花板」
所有服務都選了免費層（docs/02 有揭露），但免費層的規則是**別人訂的、會變的**。Supabase 閒置暫停、Storage 上限、Resend 每日封數——任何一個撞到都會讓網站「無聲地壞掉」，而店家可能好幾天不知道自己漏接了訂單。Phase 2 上線時，除了功能，要做一個最小監控：GitHub Actions 排程每天 ping 一次送件 API，失敗就發 email 給店家。這比任何新功能都優先。

### 2. 使用者是 C 級，你的解釋方式就是產品的一部分
他看得懂簡單程式碼，但你丟一整包 diff 他無法審。每次交付的「人工驗收三步驟」不是形式，是他唯一能真正驗證你的手段。寫驗收步驟時想像對方在手機上通勤——步驟要短、要具體、要能在瀏覽器完成。另外：**永遠不要讓他直接在 GitHub 網頁上手改程式碼來救急**，那會製造出模型不知道的變更；救急也走「開新對話 → 說明狀況 → 模型出 patch」。

### 3. 公開網站 = 你在幫他承擔真實世界的風險
Phase 2 之後這不是玩具：真人的聯絡方式存在資料庫裡、陌生人能上傳任意圖片。每次動到上傳、表單、RLS、Edge Function 的程式碼，都比動 UI 嚴重一級——套 docs/04 R3，這類改動的驗收要多加一個 T5 審查回合。另外圖片內容（版權、不當內容）的責任問題，制度只放了一句免責文案，若訂單量變大，要主動提醒使用者這塊需要人工把關或更正式的條款。

## 這套制度最可能的退化方式與預防

1. **CLAUDE.md 肥大化**：每個 session 都想把「重要的事」塞進去，兩個月後變成沒人讀完的兩百行。→ 預防：docs/06 已規定 CLAUDE.md 只放路由且改動要問使用者。看到任何 session 想往裡加內容，指向這條。
2. **handoff 斷鏈**：某次 session 中斷或偷懶沒寫 handoff，下個 session 依據過時資訊工作，錯誤開始複利。→ 預防：使用者的開場引導語（README）要求模型先讀 handoff 並**覆述目前狀態**；覆述明顯對不上 repo 實況（git log 對不上）就是斷鏈訊號，先修 handoff 再工作。
3. **規則變裝飾**：模型讀了 docs 但實際行為不理它（最常見：跳過「驗證不自驗」直接宣稱完成）。→ 預防：使用者這端的防線最有效——看到交付訊息裡沒有「CI 綠燈＋人工驗收三步驟」，直接回一句「R2」。這一個字就夠了，模型會懂。
4. **制度替代不了的東西被假裝替代了**（誠實條款）：拆解、模板、審查回合補得了執行品質；**模糊需求與品味判斷補不了**。未來的 session，當你發現自己在用很有自信的語氣講一個其實是品味的判斷——停下來，走 docs/04 R3 第 5 條：做選項，讓使用者選。承認「這題我不比你準」不丟臉，假裝準才危險。

## 交接註記（誠實揭露本次收尾的執行方式）
- 本套檔案的對抗審查在同一個對話內以「逐檔交叉核對」方式執行（檢查規則衝突、檔名引用、模糊語句），**不是**真正的 fresh-context agent——對話介面無法派 subagent。第一個能開新對話的機會，請使用者用 docs/05 T5 模板對整包 docs 做一次真正的外部審查，成本很低，值得。
- docs/03 B 節（Claude Code 模式）的模型名稱與機制未經實地驗證，使用前照 docs/06 環境核對清單過一遍。
