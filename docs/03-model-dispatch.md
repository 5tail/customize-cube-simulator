# 03 模型調度守則

> 目的：用最便宜的模型做掉最多的事，把貴的模型留給值得的地方；並且保證「做的人」和「驗的人」不是同一個腦袋。
> 本守則分兩節：A 節適用 **Claude Code 網頁版**（claude.ai/code，2026-07-04 起的預設環境）；B 節適用 claude.ai 對話介面（備援模式）。共用的升降級規則在 C 節。
> ⚠️ 使用者環境鐵則：**不碰本機**。Claude Code 的終端機版／桌面版／IDE 外掛一律不用；只用網頁版（雲端沙箱）。

## A. Claude Code 網頁版模式（預設）

雲端沙箱自動 clone repo、可改檔案、可實際執行 `npm test` / `npm run build`、完成後開 Pull Request。使用者只需在 GitHub 按 Merge。

1. **一個 session 只做一件事**。任務完成（PR 開出）就收尾。這條不因換環境而取消：長 session 一樣會 context 稀釋。
2. **預設用 Sonnet**，需要時依 C 節升級 Opus（用 `/model` 或介面上的模型選單切換；實際可用型號以介面為準，發現與本檔不符時依 docs/06 回報）。
3. **派工方式**：使用者開新 session，貼 docs/05 對應模板。Claude Code 會自動讀 repo 裡的 CLAUDE.md；模板第一行「先讀 CLAUDE.md 與 handoff」仍保留，作為雙保險。
4. **交付 = 一個 PR**，內含：程式碼改動＋**docs/07-handoff.md 的更新（同一個 PR，不可分開）**＋PR 描述中寫明人工驗收三步驟。
5. **先在沙箱驗過才開 PR**：`npm test` 與 `npm run build` 必須在沙箱實跑全綠。沙箱綠了不等於完成——repo 的 GitHub Actions 才是裁判（PR 上會自動跑測試；merge 進 main 後自動部署 Pages）。
6. **驗證不自驗**：寫程式的 session 不宣判自己通過。高風險改動（上傳、表單、RLS、Edge Function）在 merge 前，由使用者開一個**全新 Code session** 貼 docs/05 T5 模板審該 PR。
7. **雲端 session 沒有持久儲存**：每個新 session 都是白紙，handoff 制度照舊，一個字都不能省。
8. **備份規則的網頁版解釋**：純 GitHub 流程下，git 歷史即備份（每個 commit 都可還原），不再要求 `.bak` 副本；但「刪除前先問使用者」不變。

## B. 對話介面模式（備援：Code 網頁版故障或額度用盡時）

1. 調度單位是對話，調度者是使用者；預設 Sonnet，升級時機同 C 節。
2. 一個對話只做一件事；驗證不自驗（新對話貼 T5 審查）。
3. 模型在自己環境實測後，把完整檔案交給使用者，由使用者用 **GitHub 網頁上傳**（不用本機 git、不手改程式碼）。
4. 長產物落檔不落對話：超過 50 行的產出一律成檔 commit 進 repo。

## C. 升降級與重試（兩種模式共用）

| 情況 | 動作 |
|---|---|
| Haiku 級任務錯 1 次 | 直接升 Sonnet，不給第二次 |
| Sonnet 同一子任務連錯 2 次 | 帶**完整失敗軌跡**（做了什麼、錯誤訊息全文、試過什麼）升 Opus。不帶軌跡的升級無效 |
| Opus 解出來了 | 把解法寫成模式（進 docs/09-lessons.md），之後同型問題降回 Sonnet 套用 |
| 同一件事重試上限 | **2 輪**。第 3 次不是重試，是換路（docs/04 R4）或問使用者 |

## 什麼不歸這份管
何時該問使用者、何時算完成——那是判斷問題，見 docs/04。
