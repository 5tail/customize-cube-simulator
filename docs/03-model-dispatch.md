# 03 模型調度守則

> 目的：用最便宜的模型做掉最多的事，把貴的模型留給值得的地方；並且保證「做的人」和「驗的人」不是同一個腦袋。
> 本守則分兩節：A 節適用對話介面（claude.ai / Claude App，目前的主要環境）；B 節適用未來若安裝 Claude Code。共用的升降級規則在 C 節。

## A. 對話介面模式（目前預設）

對話介面沒有 subagent 可派，所以「調度」的單位是**對話（session）**，調度的人是**使用者**。模型的責任是主動建議何時該換模型、開新對話。

1. **預設用 Sonnet**。日常實作、修 bug、寫文件、照模板做事，Sonnet 全包。
2. **換 Opus 的時機**（模型發現符合任一條時，必須主動說「這題建議換 Opus 開新對話」，並先把 handoff 寫好）：
   - 架構層級的決定（動到 docs/02 的內容）
   - Sonnet 對同一個子任務連錯兩次（判定標準見 docs/04 R1）
   - 跨多檔案的大重構
   - 除錯超過 30 分鐘還在繞圈
3. **一個對話只做一件事**。任務完成就更新 handoff、結束對話。長對話 = context 稀釋 = 品質下降。訊號：對話開始重複自己、忘記前面說過的決定 → 立刻收尾換新對話。
4. **驗證不自驗**：寫程式的那個對話不能宣判自己通過。驗收的方法按順序是：
   - 機器驗：GitHub Actions 紅綠燈（建置＋測試）。
   - 新腦袋驗：開一個**全新對話**，貼 docs/05 的「審查模板」，附上要審的檔案，讓沒有包袱的 fresh context 來挑毛病。
   - 人工驗：使用者照交付時附的「人工驗收三步驟」操作。
5. **長產物落檔不落對話**：超過 50 行的產出（程式、規格）一律做成檔案（artifact/可下載檔案）並 commit 進 repo，對話裡只留結論與檔名。下個 session 讀檔案，不讀舊對話。

## B. Claude Code 模式（未來若安裝）

> 誠實標註：本節在對話介面撰寫，未實地驗證你的 Claude Code 安裝。第一次使用時先跑 `/model` 確認實際可用型號，若與下表不符，依 docs/06 更新本檔。

- 可用模型（依 2026-07 的產品線，需核對）：`claude-opus-4-8`、`claude-sonnet-4-6`、`claude-haiku-4-5`。
- **指揮官不下場**：主對話（建議跑 Opus 或 Sonnet）不做大量讀取。掃 repo、查網頁、批次改檔、跑測試，一律派 subagent（Task），主對話只收結論。
- **派工三件套**：每次派 subagent 必含（1）目標與動機（2）驗收條件（3）回報格式。缺一件就是壞派工。模板見 docs/05。
- **顯式指定模型**：批次機械工作（重命名、格式化、逐檔套用已知模式）派 Haiku；一般實作派 Sonnet；只有 C 節的升級條件成立才動用 Opus。
- **回報合約**：subagent 只回結論與 `檔案:行號`；長產物寫進檔案回傳路徑，不把全文倒回主對話。
- **驗證不自驗**：驗收派 fresh-context subagent。檔案用 read-back（重新讀出來核對），程式碼用測試或實跑，高風險判斷（會刪資料、動金流相關設定、改 RLS）加第二意見：讓兩個 agent 各自獨立作答再比對。

## C. 升降級與重試（兩種模式共用）

| 情況 | 動作 |
|---|---|
| Haiku 級任務錯 1 次 | 直接升 Sonnet，不給第二次 |
| Sonnet 同一子任務連錯 2 次 | 帶**完整失敗軌跡**（做了什麼、錯誤訊息全文、試過什麼）升 Opus。不帶軌跡的升級無效，Opus 會重踩一樣的坑 |
| Opus 解出來了 | 把解法寫成模式（寫進 docs/09-lessons.md），之後同型問題降回 Sonnet 照模式套用 |
| 同一件事重試上限 | **2 輪**。第 3 次不是重試，是換路（判定見 docs/04 R4）或問使用者 |

## 什麼不歸這份管
何時該問使用者、何時算完成——那是判斷問題，見 docs/04。
