# 06 維護協議：這套制度本身怎麼安全地更新

> 制度不更新會過時，亂更新會腐化。這份定義誰能改什麼、怎麼改。

## 權限分級

| 檔案 | 模型可自行改？ | 規則 |
|---|---|---|
| docs/07-handoff.md | ✅ 每個 session 結束**必須**改 | 照該檔內建格式 |
| docs/09-lessons.md | ✅ 踩坑後**應該**追加 | 照下方教訓格式，只增不刪 |
| docs/05 模板 | ⚠️ 可微調字句 | 三件套結構不可刪；改動要在 handoff 註明 |
| docs/00、03、04、06 | ❌ 先問使用者 | 提案格式：現行條文 → 建議條文 → 為什麼 → 風險 |
| docs/01 規格、02 架構 | ❌ 先問使用者 | 這兩份是專案的憲法 |
| CLAUDE.md | ❌ 先問使用者 | 只放路由，任何人想往裡塞內容都要抵抗（見下方腐化模式一） |

## 改檔案的通用鐵律
1. 改任何 docs/ 既有檔案前，先 `cp 檔名 檔名.bak-日期`，改完由使用者確認後才刪備份。
2. 新的長內容寫成新檔，CLAUDE.md 只加一行指向；不把長內容塞進既有檔案。
3. 每次改動在 commit message 寫明改了哪份制度、為什麼。

## 教訓怎麼寫（docs/09-lessons.md 的格式）

```
## L-編號 一句話標題（日期）
情境：出了什麼事（≤2 行）
根因：為什麼會發生（≤2 行）
規則：下次遇到 X 就做 Y（1 行，要可執行）
```

**寫規則的標準**：弱模型看得懂、做得到。「注意圖片品質」不合格；「上傳的圖在存檔前用 canvas 壓到長邊 ≤1024px」合格。

## 精簡機制（防止制度肥大）
- lessons 累積到 **30 條**時：由 Opus 開一個專門 session 做精簡——重複的合併、已寫進其他 docs 的刪除、過時的標註作廢（不直接刪，改成 `[已作廢:原因]`）。精簡結果先給使用者過目才 commit。
- handoff 只保留**最近 3 個 session** 的紀錄，更舊的剪到 docs/handoff-archive.md。
- 任何 docs 檔超過 200 行，就是該拆或該精簡的訊號。

## 環境核對清單（換環境時執行）
以下情況執行本清單：使用者安裝了 Claude Code／換了模型世代／Supabase 或 GitHub 介面大改版。
1. 核對 docs/03 B 節的模型名稱是否還存在（Claude Code 用 `/model`；對話介面問使用者目前選單上有哪些模型）。
2. 核對 docs/02 的免費層額度是否變動（查各服務官方 pricing 頁）。
3. 核對本套 docs 內所有檔名互相引用是否正確（有檔案被改名就全域搜尋修引用）。
4. 修正結果按本檔權限分級處理（多數要先問使用者）。

## 每 10 個 session 的健檢（使用者發起，貼這段給任一 Sonnet session）
```
請讀全部 docs/，回答：1. 有沒有兩條規則互相矛盾？2. 有沒有最近 5 個 session 都沒用到的規則（候選刪除）？3. lessons 有幾條、需不需要精簡？只回報，不要改。
```
