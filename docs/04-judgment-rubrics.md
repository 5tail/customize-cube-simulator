# 04 判斷力 Rubric：五個關鍵時刻怎麼判

> 這份把「需要判斷力」的時刻變成可查表的規則。每條附一個正例、一個反例。拿不準時，寧可保守（問使用者的成本遠低於做錯的成本——這位使用者是 C 級，看得懂簡單程式碼，但復原災難要靠你）。

## R1 何時該升級模型（配 docs/03 C 節使用）

**判準**：出現以下任一訊號就是「錯一次」——(a) 測試/建置紅燈且修一次後仍紅 (b) 產出和驗收條件對不上 (c) 對錯誤訊息的解釋前後矛盾。累計依 docs/03 的表。

- ✅ 正例：Sonnet 修 Three.js 貼圖方向錯誤，改了兩次 UV 座標都不對 → 這是「同一子任務連錯 2 次」，整理失敗軌跡，建議使用者換 Opus。
- ❌ 反例：Sonnet 第一次寫的表單有個錯字，自己發現自己修好 → 這不算「錯一次」，正常迭代不觸發升級。升級是給「卡住」用的，不是給「還在做」用的。

## R2 何時算真的完成（Definition of Done）

**全部打勾才算完成，缺一項就不准說「完成了」：**
1. GitHub Actions 綠燈（建置＋現有測試通過）。
2. 新功能附至少 1 個測試，或書面說明為什麼此項無法自動測試（例：純視覺效果）。
3. 附「人工驗收三步驟」：用使用者做得到的動作描述（開哪個網址、點什麼、應該看到什麼）。
4. docs/07-handoff.md 已更新。
5. 沒有偷加規格外的東西（對照 docs/01 的「明確不做」）。

- ✅ 正例：「上傳功能完成：CI 綠燈；新增 upload.test.ts 驗證檔案大小限制；人工驗收：開 https://…/，選一張 6MB 的圖，應看到『檔案過大』提示；handoff 已更新。」
- ❌ 反例：「程式碼寫好了，邏輯上應該可以動。」——「邏輯上應該」四個字出現，就代表沒完成（docs/00 洞 2）。

## R3 何時該停下來問使用者

**碰到以下任一項，停下來問，不准自行決定：**
1. 要刪除任何東西（檔案、資料表、部署、Supabase 專案設定）——即使你認為它沒用。（唯一例外：同一任務中自己建立的 .bak 備份，驗收通過後可自行清理。）
2. 任何會產生費用或動到帳號安全的操作（升級付費方案、改密碼、開新的雲服務帳號）。
3. docs/01「明確不做」清單裡的功能，或 docs/02 白名單外的技術。
4. 兩條路各有明顯代價、且影響超過一個 Phase 的取捨。
5. 品味問題（配色、文案語氣、版面）而使用者沒給過偏好 → 做 2-3 版讓使用者選。

**問的格式**：一段白話說明狀況 → 選項 A/B（各附一句代價）→ 你的建議與理由。不要只丟問題不給建議。

- ✅ 正例：「Storage 快滿了。A：現在做自動清理舊訂單圖片（要寫刪除邏輯，有誤刪風險）；B：先手動在 Supabase 後台刪（安全但你要動手）。建議 B，等訂單量穩定再做 A。你選？」
- ❌ 反例：發現舊的測試圖片佔空間，直接寫腳本全刪了，事後報告「幫你清理好了」。——刪除永遠先問（第 1 條）。

## R4 什麼訊號代表方向錯了，該換路而不是重試

**重試 = 同一方法再來一次；換路 = 承認方法錯了。出現以下任一訊號，禁止再重試：**
1. 連兩次修復都是在修「上一次修復造成的新問題」（越修越深）。
2. 為了讓方案成立，需要引入白名單外的新依賴或繞過安全機制（RLS、檔案驗證）。
3. 解釋不了為什麼上次會失敗，只是「再試一次看看」。
4. 動到的檔案數量隨重試次數增加。

**換路的動作**：先 `git checkout` 回到乾淨狀態（或還原備份），寫下失敗軌跡，然後：換一個根本不同的做法，或依 R1 升級，或問使用者。

- ✅ 正例：為修手機上的 3D 效能，第一次改渲染參數沒用，第二次想引入新的 WebGL 函式庫 → 觸發訊號 2，停止。回報：「效能問題可能出在貼圖解析度而非渲染器，建議改成上傳時壓縮圖片，這不需要新依賴。」
- ❌ 反例：CI 紅燈，連改五次設定檔，每次都「應該是這個問題」，repo 裡多了三個試驗用檔案沒清。——訊號 3＋4 早就亮了。

## R5 品質底線怎麼驗（最低標準，不是最高標準）

**每次 commit 前自查：**
1. 密鑰掃描：搜尋 diff 裡有沒有 `key`、`secret`、`token`、`password` 的實際值。有 → 絕對不准 commit。
2. `npm run build` 在本地（或 CI）通過。
3. 改動的檔案裡沒有殘留的 debug 輸出、註解掉的大段舊碼、TODO 沒登記進 handoff。
4. 使用者上傳內容的路徑（上傳、表單）都有驗證：型別、大小、頻率。

- ✅ 正例：commit 前發現測試時把 Resend key 寫死在檔案裡試寄信 → 移到環境變數、確認 git 歷史沒進去過，才 commit。
- ❌ 反例：「先 commit 上去看 CI 過不過」把含密鑰的檔案推上公開 repo。——公開 repo 的 git 歷史等於全世界都看得到，事後刪檔案救不回來，只能作廢密鑰。

## 這份 rubric 的極限（誠實條款）
R1–R5 管得住流程品質，管不住「這個產品好不好用」。使用體驗、視覺品味、商業判斷（報價欄位該有哪些）——遇到就走 R3 第 5 條：做選項、給建議、讓使用者決定。不要用 checklist 的自信去掩蓋品味的不確定。
