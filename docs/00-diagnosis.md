# 00 現況診斷：這個工作模式最容易出錯的三件事

> 讀者：未來所有 Claude session（Sonnet/Opus）。這份是整套制度的「為什麼」，其他檔案的規則都是為了修這三個洞。
> 使用者環境：對話介面（Claude 桌面 App / claude.ai 網頁 / 手機）跨裝置切換，專案唯一落腳點是 GitHub。使用者程式程度：看得懂簡單程式碼、改過別人的東西（C 級）。

## 洞 1：Session 之間完全失憶（最嚴重）

**症狀**：對話介面每次開新對話都是白紙。上次決定的架構、改到一半的功能、踩過的坑，下一個 session 全部不知道，會重新發明、重複踩坑、甚至推翻已定案的決定。

**修法（已制度化）**：
1. 唯一事實來源是 GitHub repo 裡的 `docs/` 資料夾，不是任何一段對話。
2. 每個 session 結束前，必須更新 `docs/07-handoff.md`（格式見該檔），寫「做到哪、下一步、未解問題」。
3. 每個 session 開場，使用者貼固定引導語（見 README），模型第一件事是讀 CLAUDE.md 和 handoff。
4. 「對話裡講好但沒寫進檔案的決定」視同不存在。任何決定要生效，必須落檔。

## 洞 2：模型看不到真實世界，卻假裝看得到（最危險）

**症狀**：對話介面裡的模型碰不到你的部署網站、跑不了你的瀏覽器，卻常說「這樣就能動了」。程式碼看起來對和實際能跑是兩回事，弱模型特別容易自己宣稱驗收通過。

**修法（已制度化）**：
1. 驗證外包給機器：GitHub Actions 每次 push 自動跑測試與建置，紅綠燈是唯一裁判，模型的「應該可以」不算數。
2. 模型交付任何程式碼，必須附「人工驗收三步驟」：用你（C 級使用者）做得到的動作寫，例如「打開網址 → 上傳一張 jpg → 應該看到方塊六面出現圖片」。
3. 判斷規則見 `docs/04-judgment-rubrics.md` 的「何時算真的完成」。

## 洞 3：範圍蔓延與技術棧膨脹（最花錢花時間）

**症狀**：模型愛炫技。你要一個上傳表單，它順手加了會員系統；你要修一個 bug，它重構了整個資料夾；每次換 session 就多裝一個新套件。三個月後專案沒人看得懂，包括模型自己。

**修法（已制度化）**：
1. `docs/02-architecture.md` 是技術白名單。清單外的框架、服務、重大依賴，一律先問使用者才能加。
2. 每個任務用 `docs/05-prompt-templates.md` 的模板派工，模板強制寫明「不做什麼」。
3. 改既有檔案前先留備份（規則在 CLAUDE.md 鐵律）。

## 誠實條款：這套制度補不了的事

- **品味與模糊判斷**：UI 好不好看、報價流程順不順手，checklist 補不了。遇到時的正確動作是：做 2-3 個版本讓使用者選，或明說「這是品味問題，請你決定」，不要假裝有標準答案。
- **本診斷的產生環境限制**：本檔在 claude.ai 對話介面產生，無法實地檢查使用者未來的 Claude Code 設定。若未來安裝 Claude Code，第一個 session 應執行 `docs/06-maintenance.md` 的「環境核對清單」修正本套檔案中的假設。
- **免費額度是天花板不是無限**：見 `docs/08-letter.md` 第一件事。
